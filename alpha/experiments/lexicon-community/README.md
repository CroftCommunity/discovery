# lexicon-community — RUN-LEX-01

A clean-room, second-implementation spike engaging the Lexicon Community
attestation work: the published **CID-First Attestation Specification**
(badge.blue, Nick Gerakines) and the **lexicon.community** calendar / RSVP /
location lexicons. The engagement brief is `../attest-family/ENGAGE-LEX.md`.

The wager (brief §0): the attestation conversation already exists with a
published spec at its center, so the winning contribution is not a rival proposal
but **an independent implementation, an adversarial fixture corpus, and the one
missing layer** (freshness/status without verifier callback). This lane builds
all three, red-first.

## Headline results

- **Interop**: our clean-room verifier accepts records signed by the **reference
  implementation** (inline + remote) — two independent codebases agreeing on the
  same bytes. The interop probe also pinned the spec's underspecified signing
  input empirically (36-byte binary CID, SHA-256 ECDSA, low-S — AMBIGUITIES A-4/A-5).
- **Ground truth**: our DAG-CBOR + CID path is **byte-identical and CID-identical**
  to real records pulled from public PDSes (EXP-LEX-02).
- **A finding with teeth** (A-1): inline attestations authenticate *a key*, not
  *the issuer*, until the verifier checks the key against the issuer DID document —
  we demonstrate the forgery and ship the fix (strict posture, default on).
- **The missing layer** (EXP-LEX-04): holder-stapled status verifies freshness and
  revocation with **zero callback to the issuer** — the OCSP privacy lesson applied
  to attestations. 870 B staple, offline verify.
